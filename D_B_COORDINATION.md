# D 与 B 模块协作说明

> 目标：在不等待 C 社交模块的前提下，让“用户、数据与质量”（D）能够为“活动创建、详情与报名闭环”（B）提供一致的数据规则、隐私保护和可回归测试能力。

## 一、当前边界

- B 负责活动详情、创建、报名、审批、退出、取消，以及这些动作在页面上的状态展示。
- D 负责类型定义、本地存储、用户资料与偏好、信用分、演示数据、数据恢复、质量回归。
- `lib/types.ts`、`lib/storage.ts`、`components/app-provider.tsx` 是共享文件。任何接口变更需先在 Issue 或 PR 中说明，并由双方同步后再合并。

## 二、已完成的 B 接口（由 D 集成实现）

以下接口现已落地，可供 D 后续质量工作和自动化测试使用：

- 活动增加 `endTime`，并以 `cancelled > finished > full > open` 同步状态。
- 创建活动可填写结束时间；详情页展示结束时间、已结束和取消原因。
- 发起人取消活动必须填写原因；待审核申请支持撤回。
- 详情页与成员列表已按 `profileVisible` 脱敏。
- 创建、申请、退出、取消、审批、拒绝、撤回按钮已具备约定的 `data-testid`。

## 三、B 接口说明与后续扩展

### 1. 活动生命周期

`Activity` 现已包含 `startTime`、`endTime` 和 `status`。到达结束时间后，活动自动进入 `finished`；旧本地数据缺少结束时间时，数据层会以开始时间后两小时补齐。状态优先级统一为：

```text
cancelled > finished > full > open
```

| 状态 | 触发条件 | B 页面行为 | D 数据行为 |
| --- | --- | --- | --- |
| `open` | 未取消、未结束且未满员 | 可以申请 | 正常保存、允许创建申请 |
| `full` | 已加入人数达到上限 | 显示满员，不允许新申请 | 审批通过后同步状态 |
| `finished` | 达到统一结束条件 | 禁止申请、退出和审批 | 用于活动历史、完成率和信用分 |
| `cancelled` | 发起人完成取消操作 | 显示取消原因，禁止业务操作 | 通知成员并保留记录 |

### 2. 取消、退出与撤回申请

B 需明确以下动作的字段与入口；D 才能计算信用分、历史状态和通知：

| 动作 | B 需要完成 | D 的后续处理 |
| --- | --- | --- |
| 取消活动 | 发起人填写 `cancelReason`；已结束活动不可取消 | 通知成员、写入历史、按规则调整信用分 |
| 成员退出 | 判断是否允许已结束活动退出；提供退出时间 | 依据距离活动开始/结束的时间计算信用分 |
| 撤回申请 | 为 `ApplicationStatus = "withdrawn"` 提供实际操作 | 保留申请记录，避免重复申请状态混乱 |
| 编辑活动 | 明确哪些字段可编辑，特别是时间、地点、人数 | D 根据变更类型发送通知、处理状态重新计算 |

当前已接入字段：

```ts
interface Activity {
  endTime: string;
  cancelReason?: string;
  updatedAt?: string;
}

interface Application {
  withdrawnAt?: string;
}
```

### 3. 资料隐私展示

D 已提供 `User.profileVisible`。B 在活动详情和成员列表中需要按如下规则渲染：

| 查看者 | 被查看用户设置 | 显示内容 |
| --- | --- | --- |
| 本人 | 任意 | 完整资料 |
| 他人 | `profileVisible: true` | 昵称、学院、年级、兴趣、简介 |
| 他人 | `profileVisible: false` | 昵称和默认头像；隐藏学院、年级、兴趣、简介 |

建议 B 将成员资料展示抽成一个组件或辅助函数，避免详情页与组队页出现不同的脱敏规则。

### 4. 可测试的稳定交互点

D 会补 Playwright 核心流程测试。B 需要为关键控件提供稳定的 `data-testid`，不要让测试依赖中文文案、图标或 Tailwind 类名。

```text
activity-create-submit
activity-apply-submit
activity-exit
activity-cancel
application-approve-{applicationId}
application-reject-{applicationId}
application-withdraw-{applicationId}
```

每个动作应同时保证：成功后页面状态更新；失败时显示可读的中文错误提示；刷新后本地数据仍一致。

## 四、D 在收到接口后的实施项

1. 在 `lib/types.ts`、`lib/storage.ts` 和种子数据中接入 B 确认的字段。
2. 统一计算活动生命周期，并在读取、创建、审批、退出、取消、编辑后同步状态。
3. 按已确认的规则记录信用分变动和成就。
4. 更新个人中心的活动历史、完成率、信用分说明。
5. 更新导入/导出数据校验，使其包含 B 新增字段。
6. 编写并执行“创建 → 申请 → 审批 → 退出/取消/结束 → 重置”的端到端回归测试。

## 五、建议协作顺序

1. B、D 在无痕窗口共同走一次 `DEMO_SCRIPT.md` 和 `QUALITY_CHECKLIST.md`。
2. D 基于当前测试选择器补齐自动化回归。
3. 双方分别执行 `npm.cmd run lint`、`npm.cmd run build`，在 PR 中记录结果和未完成事项。

## 六、最小验收清单

- [ ] 已结束或已取消活动不能继续申请、审批或退出。
- [ ] 取消、退出、撤回申请的历史状态可刷新后保留。
- [ ] 关闭公开校园名片后，另一账号无法在详情或成员列表看到私密资料。
- [ ] 完成活动状态变化后，个人中心的完成率、信用分和活动历史保持一致。
- [ ] 关键按钮具备约定的 `data-testid`，核心流程自动化测试可稳定运行。
