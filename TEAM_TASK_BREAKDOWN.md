# CampusMate 四人开发分工与功能任务书

> 项目定位：面向校园场景的“学习、运动、吃饭、活动”搭子匹配平台。
>
> 本项目用于暑期学校展示，优先完成可演示、可交互、视觉完整的前端 MVP。所有数据可以使用本地 Mock 数据，不要求真实上线、真实支付、真实即时通讯或真实地图服务。

## 一、最终展示目标

演示者可以用一条完整路径展示产品价值：

1. 访客打开首页，看到校园活动和搭子推荐。
2. 使用搜索、分类、时间、地点和标签筛选活动。
3. 打开活动详情，查看发起人、人数、时间、地点和规则。
4. 登录或注册一个演示账号。
5. 加入活动，看到名额和参与状态发生变化。
6. 创建一条“周末羽毛球 / 图书馆自习 / 食堂拼饭”等活动。
7. 在“我的活动”和“我的组队”中查看自己参加或发起的内容。
8. 查看个人资料、兴趣标签、信用分和历史活动。
9. 用演示数据展示通知、收藏、评价和举报等完整产品闭环。

## 二、统一技术与协作约定

### 技术栈

- Next.js 16 + React 19 + TypeScript
- Tailwind CSS
- 本地 Mock 数据或浏览器端状态模拟后端
- GitHub + Git 分支 + Pull Request
- Codex 负责编码、重构、测试和文档；成员负责需求确认、验收和代码 Review

### 分支规则

基础分支：`main`

四条长期功能分支：

```text
feature/discover-ui        # 成员 A：发现页与视觉体验
feature/activity-flow      # 成员 B：活动创建、详情、报名流程
feature/team-social        # 成员 C：组队、消息、通知、社交互动
feature/profile-quality    # 成员 D：个人中心、数据、测试与演示
```

每个成员从最新 `main` 创建自己的分支，禁止直接提交 `main`。每项功能完成后提交 PR，至少由另一名成员 Review 后再合并。

### Commit 约定

```text
feat: add activity filter panel
fix: correct activity join status
ui: improve mobile activity card
test: add activity flow checks
docs: update demo guide
```

## 三、四人详细分工

## 成员 A：发现页、首页与设计系统

### 负责范围

负责用户打开产品后看到的第一印象，以及所有活动的浏览、搜索和筛选体验。

### 必做功能

- 首页 Hero 区域：产品口号、核心 CTA、今日活动数量
- 活动卡片：封面、标题、分类、时间、地点、已报名人数、剩余名额
- 活动分类：自习、运动、拼饭、社团、比赛、出游、其他
- 关键词搜索：标题、描述、地点、标签模糊匹配
- 多条件筛选：分类、日期、时间段、地点、人数范围
- 排序：即将开始、最热门、最新发布、距离优先的模拟排序
- 推荐活动区：根据兴趣标签生成本地推荐
- 热门榜单：热门活动、活跃发起人、校园趋势标签
- 空状态、加载状态、错误状态、无结果状态
- 响应式布局：桌面端、平板端、手机端
- 顶部导航、底部导航、移动端菜单
- 统一颜色、字体、按钮、标签、卡片、表单、弹窗样式

### 可选增强功能

- 收藏活动按钮
- 最近浏览记录
- “只看还可报名”开关
- 推荐理由，例如“与你的羽毛球兴趣匹配”
- 活动地图缩略图的静态占位卡
- 骨架屏和页面切换动画

### 主要交付文件

```text
app/page.tsx
components/activity-card.tsx
components/search-filter.tsx
components/navigation.tsx
components/ui/*
app/globals.css
```

### 验收标准

- 首页首次打开 3 秒内能理解产品用途。
- 至少有 6 条不同类型的活动卡片。
- 搜索、分类、日期筛选可以组合使用。
- 在 390px 手机宽度下不出现横向溢出。
- 无结果时有明确提示和清除筛选入口。

## 成员 B：活动创建、详情与报名闭环

### 负责范围

负责“活动从创建到结束”的完整业务流程，是演示中最重要的可交互模块。

### 必做功能

- 活动详情页
- 活动基本信息：标题、简介、分类、标签、时间、地点、人数上限
- 发起人信息：头像、昵称、学院、信誉分、历史活动数
- 活动状态：报名中、即将开始、进行中、已结束、已取消
- 创建活动表单
- 表单校验：标题长度、时间不能早于当前时间、人数范围、必填字段
- 编辑自己创建的活动
- 取消自己创建的活动
- 报名参加活动
- 取消报名
- 满员状态与候补状态
- 报名成员列表
- 发起人审核成员的模拟状态
- 活动规则和注意事项
- 活动时间线：创建、报名、开始、结束
- 活动详情分享按钮（复制链接的模拟提示）

### 可选增强功能

- 报名问题：年级、运动水平、是否自带装备
- 候补转正模拟
- 活动签到按钮
- 活动结束后自动进入待评价状态
- 活动模板：一键再次发布相似活动
- 创建草稿和预览功能

### 主要交付文件

```text
app/activity/[id]/page.tsx
app/create/page.tsx
components/activity-detail.tsx
components/activity-form.tsx
components/member-list.tsx
lib/activity-actions.ts
lib/validation.ts
```

### 验收标准

- 能从首页进入详情页，再完成报名和取消报名。
- 能创建一条活动，并在首页或“我的活动”中看到。
- 错误输入会显示中文提示，不能提交无效表单。
- 满员、已结束、已取消活动的按钮状态正确。
- 刷新页面后演示数据仍能恢复到稳定状态，不能出现白屏。

## 成员 C：组队、消息与社交互动

### 负责范围

负责“人与人连接”的体验，让项目从活动列表升级为真正的校园搭子产品。

### 必做功能

- 我的组队页：我参加的活动、我发起的活动、候补活动
- 活动成员分组展示
- 搭子卡片：昵称、头像、学院、兴趣、匹配度、在线状态
- 兴趣标签匹配：根据共同兴趣计算模拟匹配度
- 组队邀请：邀请某位同学加入活动的模拟流程
- 接受 / 拒绝邀请
- 站内通知中心
- 通知类型：报名成功、活动提醒、邀请、系统消息
- 未读数量角标和全部标记已读
- 活动评论区
- 评论发布、删除自己的评论
- 收藏和取消收藏活动
- 举报活动或用户的模拟弹窗

### 可选增强功能

- 私聊页面的演示版消息流
- 快捷消息： “你好，我也是初学者”“一起去图书馆吗？”
- 共同兴趣标签高亮
- 组队完成度进度条
- 组队成员投票选择时间或地点
- 活动群公告
- 屏蔽用户的本地模拟状态

### 主要交付文件

```text
app/my-team/page.tsx
app/notifications/page.tsx
app/messages/page.tsx
components/team-card.tsx
components/notification-list.tsx
components/comment-list.tsx
lib/social-actions.ts
```

### 验收标准

- 加入活动后可以在“我的组队”看到对应活动。
- 通知中心至少能展示 4 种通知。
- 评论、收藏、邀请操作有明确成功反馈。
- 组队页必须包含空状态和已有数据两种状态。
- 不实现真实 WebSocket，但交互效果要像真实产品。

## 四、阶段计划

### 阶段 0：基础框架确认（第 1 天）

所有成员共同完成：

- Review 基础 PR
- 合并到 `main`
- 安装依赖并确认本地启动成功
- 阅读 `README.md`、`AGENTS.md`、`CONTRIBUTING.md`
- 确认数据模型、页面路由和视觉基调
- 每人从 `main` 创建自己的功能分支

交付物：四条功能分支均能启动项目，且没有未提交的本地修改。

### 阶段 1：核心页面和数据闭环（第 2-3 天）

- A：完成首页、活动卡片、搜索筛选
- B：完成活动详情和创建表单
- C：完成我的组队基础页
- D：完成登录、注册、Mock 数据和状态管理

交付物：用户能浏览活动、登录、查看详情、创建活动。

### 阶段 2：核心互动功能（第 4-5 天）

- A：完成响应式适配、推荐区和状态页面
- B：完成报名、取消报名、满员和取消活动
- C：完成通知、评论、收藏和邀请
- D：完成个人中心、兴趣标签和统计数据

交付物：用户能完成“浏览—登录—查看—报名—组队—查看个人数据”流程。

### 阶段 3：体验增强和质量保障（第 6 天）

- A：统一视觉、动效、空状态和细节
- B：补齐表单校验和异常状态
- C：补齐社交反馈和举报流程
- D：进行全流程测试、修复构建问题、整理演示数据

交付物：所有分支分别提交 PR，完成交叉 Review。

### 阶段 4：集成与最终展示（第 7 天）

- 按 A → B → C → D 顺序合并 PR
- 解决冲突并统一命名、样式和 Mock 数据
- 执行 `npm run lint`
- 执行 `npm run build`
- 使用干净浏览器完成演示
- 准备截图、产品介绍和 Codex 开发过程说明

交付物：`main` 分支稳定，能在 5 分钟内完整展示核心功能。

## 六、统一数据模型建议

### User

```ts
type User = {
  id: string;
  name: string;
  avatar: string;
  college: string;
  grade: string;
  bio: string;
  interests: string[];
  creditScore: number;
  joinedCount: number;
  hostedCount: number;
};
```

### Activity

```ts
type Activity = {
  id: string;
  title: string;
  description: string;
  category: string;
  tags: string[];
  startTime: string;
  endTime: string;
  location: string;
  maxMembers: number;
  memberIds: string[];
  hostId: string;
  status: "open" | "full" | "ongoing" | "finished" | "cancelled";
  cover: string;
};
```

### 其他实体

- `Notification`：通知中心
- `Comment`：活动评论
- `Invitation`：组队邀请
- `Favorite`：收藏关系
- `Report`：举报记录
- `Review`：活动结束后的评价

## 七、交叉 Review 分配

为避免“自己写、自己合并”，采用固定 Review 关系：

| 提交人 | 第一 Review 人 | 重点检查 |
|---|---|---|
| A | B | 页面交互、活动数据使用 |
| B | C | 报名流程、异常状态 |
| C | D | 状态一致性、通知数据 |
| D | A | 视觉统一、用户流程 |

每个 PR 至少确认：

- 功能是否符合任务书
- 是否影响其他模块
- 是否有空状态和错误状态
- 是否适配移动端
- 是否通过 `npm run lint`
- 是否需要更新 README 或演示脚本

## 八、最终演示脚本

### 场景 1：找自习搭子

首页 → 搜索“图书馆” → 选择“自习” → 打开活动详情 → 登录 → 报名 → 查看我的组队。

### 场景 2：发布羽毛球活动

登录 → 发布活动 → 填写时间、地点、人数和标签 → 提交 → 首页查看新活动 → 其他账号报名。

### 场景 3：组队互动

查看活动成员 → 发送邀请 → 通知中心出现邀请 → 接受邀请 → 评论区留言 → 收藏活动。

### 场景 4：个人成长

进入个人中心 → 查看参与次数、发起次数、信用分和兴趣标签 → 修改标签 → 返回首页查看推荐变化。

## 九、明确不做的内容

为了保证展示稳定，以下内容不作为本轮必做：

- 真实手机号、邮箱或第三方登录
- 真实云数据库和服务端鉴权
- 真实即时通讯和 WebSocket
- 真实支付、校园卡或费用结算
- 真实地图导航和定位权限
- 真实短信、邮件或推送服务
- 复杂的后台权限系统
- 生产环境部署和长期运维

这些功能可以在汇报中作为“后续商业化和上线阶段规划”展示。
